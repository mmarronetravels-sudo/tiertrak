OPEN ITEMS / FOLLOW-UPS
  - [held from PR body] requireAuth is duplicated verbatim between
    middleware/authorizeInterventionAccess.js (Session 25) and
    middleware/authorizeDocumentAccess.js (this PR). Intentional
    DRY debt to keep each resource-scoped authorizer independently
    reviewable. Target: chore/ branch extracting requireAuth into a
    shared middleware/requireAuth.js module and updating both
    resource files to import from it. No behavior change.

  - [held from PR body; security WARN 1] routes/studentDocuments.js
    POST /upload middleware order is requireAuth → multer →
    requireDocumentUploadAccess. Authenticated-but-unauthorized
    callers buffer up to 25 MB in memory before the authorizer
    rejects them. Low-privileged accounts can spam the route to
    force repeated 25 MB allocations and get 403. No rate limiter
    is bound to this route (project-wide gap). Target: sec/ branch
    adding a busboy-level student_id peek before multer, or
    express-rate-limit on the upload route, or both.

  - [held from PR body + privacy WARN 1] docs/ai-context/PRIVACY_REVIEW.md
    FERPA inventory does not yet declare student_documents as a PII
    surface. Deferred to a follow-on docs commit per the precedent
    already set in that file. When the doc surface is declared,
    also record the Parent-delete-of-own-upload retention posture
    (per privacy-reviewer INFO 2).

  - [new; tenant-auditor WARN 1] routes/studentDocuments.js
    GET /student/:studentId SELECT is not join-bound to
    students.tenant_id. Functionally safe today (middleware resolves
    the student's tenant + the SELECT has AND sd.tenant_id = $2)
    but asymmetric with the defense-in-depth JOIN the same PR
    added to GET /expiring. Target: small chore/ branch adding
    JOIN students s ON sd.student_id = s.id AND s.tenant_id = sd.tenant_id
    to both parent and staff branches of the SELECT.

  - [new; security WARN 2] jwt.verify(token, process.env.JWT_SECRET)
    is called without an algorithms allowlist at 6 call sites:
    routes/auth.js:166, 294, 572; routes/tier1-assessments.js:47;
    middleware/authorizeInterventionAccess.js:24;
    middleware/authorizeDocumentAccess.js:41. Residual
    algorithm-confusion exposure. Target: single sec/ sweep pinning
    every jwt.verify to { algorithms: ['HS256'] } plus a paired
    signing-side audit. Related master-index Followup 57
    (sec/require-auth-on-unprotected-routes) — this is a natural
    second commit on that branch.

  - [new; tenant-auditor WARN 2] student_documents has independent
    FKs on tenant_id and student_id, not a composite FK. Nothing at
    the schema layer forces student_documents.tenant_id to equal
    the referenced student's tenant_id; every row-level constraint
    is enforced in application code. Target: future migration/
    branch making students have UNIQUE (id, tenant_id) and changing
    the student_documents FK to (student_id, tenant_id) REFERENCES
    students(id, tenant_id). Makes cross-tenant drift impossible at
    the DB layer. Larger blast radius than the other followups.

  - [new; privacy INFO 3 — minor] routes/studentDocuments.js
    SELECT sd.* on GET /student/:studentId and GET /expiring
    returns student_documents.file_url, which is the raw S3 HTTPS
    URL written at upload time. Since download is presigned-only,
    this column is unused by the client. Candidate for a chore/
    branch that either stops selecting it from list responses or
    deprecates the column entirely. Minor data-minimization
    cleanup, lowest priority of the six.
