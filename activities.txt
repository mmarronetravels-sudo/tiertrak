CATCHES (Cowork-side and Claude Code-side observations worth banking)

  1. Provisioning preflight discipline. The seed script's three
     RAISE EXCEPTION preflights (subdomain unique, email unique,
     intervention-template name match) caught the first
     provisioning attempt's template-name mismatch cleanly: the
     entire transaction rolled back with zero partial state. No
     manual cleanup needed; just regenerate the roster with valid
     template names and retry. This is the script's atomic-
     transaction design working as intended.

  2. Roster template names must match production intervention_
     templates exactly. First attempt used 4 plausible-sounding
     names (Reading Intervention, Math Intervention, Behavior
     Support Plan, Check-In/Check-Out (CICO)) that don't exist
     in the production system-default bank. Production has 70
     templates with very specific names (e.g., the actual entry
     is "Check-in/Check-out" with lowercase "in" and no
     parenthetical). For future tenant provisionings, query
     intervention_templates WHERE tenant_id IS NULL ORDER BY name
     before drafting roster INTERVENTIONS to avoid this loop.

  3. Schema knowledge gaps from Cowork-side cost iteration. Two
     verification queries failed mid-session due to wrong column
     name assumptions: student_interventions has no tenant_id
     column (scoping is via student_id → students.tenant_id);
     student_interventions's template FK column is
     intervention_template_id, not template_id. Both surfaced via
     ERROR responses and were quickly corrected. Worth banking:
     when writing verification queries against tenant-scoped
     tables, prefer JOINs through the canonical tenant-scoping
     column rather than assuming a redundant tenant_id on every
     table.

  4. Seed script reference resolution by name is fragile. The
     seed script's INSERT INTO student_interventions and
     PROGRESS_NOTES statements resolve student references by
     (first_name, last_name) WHERE clauses rather than by
     external_id (or by tracking the INSERT...RETURNING student
     ids in a JS-side map). Synthetic rosters at any reasonable
     scale will have name collisions (the 270-student roster
     produced 2 duplicate name pairs by chance via the random
     name generator). When a name collides, the script inserts
     the same intervention/note against EVERY matching student
     — which surfaced as +2 interventions and +5 progress notes
     attached to Tier 1 students that should have had none.
     Cleanup was straightforward (tenant-scoped DELETE on T1
     students for those two tables) and the underlying tenant
     scoping was correct. But the script should be hardened:
     resolve references by external_id (tracked via INSERT...
     RETURNING) rather than by name lookup. Tracked as
     Followup #33.

  5. CC chat rendering and password fidelity. The original
     prospect provisioning's plaintext passwords (output by the
     seed script via CC's bash tool, captured as part of CC's
     tool result, then rendered into CC chat) did not bcrypt-
     verify against the corresponding hashes in the database.
     CC's standalone test on a tiny isolated roster showed all
     plaintext + hash pairs verifying correctly — the script's
     emit logic is provably sound. So the corruption happened
     between the script's stderr and the operator's clipboard
     (the chat-rendering layer). Most likely candidates: markdown
     processing in the chat UI transformed a base64url character
     (- ↔ em-dash, smart-quote substitution); column-alignment-
     induced copy-paste error; or some other subtle character-
     substitution in the rendered table. Recovery worked
     around the issue cleanly: a rotation script was run in the
     operator's local terminal (NOT via CC's bash tool), so
     plaintexts went directly to terminal scrollback (byte-
     accurate); UPDATE statements via psql replaced the original
     hashes; logins verified post-rotation. Future password-
     handling work follows the same rule: seed scripts run in
     the operator's terminal, never via CC's bash tool. Tracked
     as Followup #34.

  6. Cowork-side miss on the password-handling channel. The
     initial seed run for the new prospect was via CC's bash
     tool, which placed plaintexts in the chat transcript. CC
     flagged this correctly at the time ("Plaintext passwords
     are now in this conversation transcript. That's an inherent
     side effect of running the script via the Bash tool").
     Cowork-side did not redirect the seed script run to the
     operator's local terminal at that point — that decision
     would have prevented both the rotation requirement AND the
     chat-rendering mismatch issue downstream. For Session 41+:
     the SMOKE_TESTING.md walkthrough (Followup #11) and any
     future password-related operational doc must lead with
     "run in your local terminal, never via CC's bash tool."

  7. Atomic transaction recovery works as designed. Three
     transactions were attempted today (initial seed, cleanup,
     password rotation) and one of them rolled back cleanly on a
     preflight RAISE EXCEPTION. Zero partial-state cleanup
     needed across the day. The DO-block + BEGIN/COMMIT envelope
